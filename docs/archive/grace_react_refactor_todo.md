# GRACE を ReAct 化するリファクタの進め方 — TODO

本ドキュメントは、GRACE 自律エージェントを「静的・線形な Plan-Execute ループ」から
「観測駆動の ReAct（Reason–Act–Observe）ループ」へ移行するためのリファクタ計画である。

## 0. 結論（先に要点）

「ReAct 化」は単独では成立しない。**較正された Confidence が ReAct ループの操縦桿
（継続 / 分岐 / 介入 / 打ち切り）を駆動する**ため、その前提となる測定基盤（S0）と
較正（S1）を先に置く必要がある。

依存順は **S0 評価ハーネス → S1 Confidence の groundedness 化＋較正 → S3 ハイブリッド ReAct**。
「測れるようにしてから直す」を原則とし、各段に**測定可能な完了条件（Definition of Done）**を置く。

| フェーズ | 内容 | 規模感 | DoD |
|---|---|---|---|
| **S0** | 評価ハーネス整備＋ベースライン測定 | 約1〜2日 | スコア表が出る |
| **S1** | Confidence を較正された根拠妥当性へ | 約2〜4日 | ECE がベースライン改善 |
| **S3** | ハイブリッド ReAct スケルトン | 約1〜2日 | 正解率が静的版以上 |

## 解説：なぜこの設計・この順か

### 1. ReAct 化の本質
静的計画は「やってみる前」に step 列を固定する。しかし RAG の検索結果を見るまで本来は
次手を決められない（検索で前提が崩れたら分解方針ごと変えるべき）。ReAct は各ターンで
観測してから行動を選ぶので、想定外入力と分岐に強くなる。

### 2. なぜ「ハイブリッド」か
完全 ReAct への全面移行は移行コストが高く回帰リスクも大きい。現行 Plan を**初期仮説**として
保持し、各ターンで妥当性を再判定する形なら、既存の executor 資産をほぼそのまま使え、
単純質問は現行パスのまま安全に温存できる。

### 3. なぜ Confidence(S1) が ReAct(S3) より先か
ReAct ループの分岐（継続 / 切替 / 介入 / 打ち切り）は**すべて confidence が駆動**する。
未較正のスコアで分岐すると「動くが正しさは未検証」の PoC 止まり。較正済みにして初めて
分岐の正しさが担保される。

### 4. なぜ評価(S0) が最初か
S1 の較正は「正解付きデータで confidence → 実正解率を測る」のが前提＝S0 が完成していないと
着手できない。また ReAct 化の効果（正解率が静的版以上か）も S0 が無ければ判定不能。
ゆえに「**測れるようにしてから直す**」順が安全。

### 5. S3 が意外と軽い理由
新規実装は実質 `_decide_next_action` ＋ Scratchpad スキーマ ＋ ループ骨格のみ。
ツール実行・タイムアウト・介入・ask_user は `executor.py` に既存のものを再利用できる。

より広いスコープ（全体分析）では、以下も射程に入る。
